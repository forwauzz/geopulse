# Make Something Agents Want

If you want the useful version fast, here it is: agents do not care how clever your homepage sounds. They care whether your site, docs, and product surface are easy to parse, compare, and act on. That is why "make something agents want" is starting to matter as much as "make something people want."

The idea showed up clearly in YC's February 21, 2026 Lightcone episode about the agent economy. The useful takeaway is not that humans stopped mattering. It is that agents are increasingly doing the research, implementation, and recommendation work that used to happen entirely in a human browser session.

## What "make something agents want" actually means

It does not mean designing for robots instead of people.

It means recognizing that a growing amount of software discovery now happens through agent-mediated workflows:
- a coding assistant chooses a library
- an implementation agent compares docs
- an AI workflow tool decides which integration is easier to use
- a buyer or operator asks a model which tool fits a job

In those moments, your site is not being judged by visual taste first. It is being judged by legibility.

## Why documentation is now part of your acquisition surface

For a long time, teams treated docs as support material.

That model is weakening.

Now documentation often acts as:
- product explanation
- implementation proof
- trust surface
- comparison input
- machine-readable acquisition layer

If the docs are clear, structured, and direct, an agent can reuse them.

If they are vague, fragmented, or hidden behind marketing language, an agent has less to work with. That does not always mean the product loses. But it does mean the product is harder to choose.

## Why Supabase is a useful example

Supabase is a good example not because it is the only company with strong docs, but because the docs make the product easy to understand in operational terms.

On the public docs surface, Supabase does a few simple things well:
- it explains the product in clear product categories
- it exposes reference material separately from narrative guides
- it gives explicit API and implementation paths
- it keeps examples close to the explanation
- it makes the docs navigable as a working system, not just a content library

You can see that directly on the docs home, the reference section, and the REST API guide:
- https://supabase.com/docs
- https://supabase.com/docs/reference
- https://supabase.com/docs/guides/api

That matters because an agent evaluating a tool does not need brand theatre first. It needs answers to practical questions like:

1. What is this product?
2. What can it do?
3. How do I use it?
4. Where is the API or implementation path?
5. Can I trust the docs enough to proceed?

Supabase's docs do a good job of answering those questions without making the reader fight for the answer.

## What this means for your own site

Most companies do not need a giant developer-docs portal to benefit from this idea.

They do need pages that make the business easy to interpret.

That usually means:
- the core problem is named clearly
- the answer appears early
- claims are supported where support is needed
- important capabilities are stated plainly
- related pages connect into a coherent topic path
- the page can stand on its own when quoted out of context

This is the same reason GEO-Pulse keeps pushing on crawlability, extractability, structure, and trust together. Agents do not only need access. They need usable source material.

## What agents do not want

Agents do not want:
- vague category language
- buried implementation details
- glossy pages with no operational explanation
- PDFs as the primary source of truth
- pages where the meaning depends on video alone
- pages where every important claim requires heavy inference

Humans do not love those things either. Agents are just less patient with them.

## Do images and video still help

Yes, if they support the page instead of carrying the page.

Images can help when they:
- show a real interface
- explain a workflow
- annotate an example
- make a concept easier to grasp quickly

Video can help when it:
- demonstrates the product
- gives context for a complex explanation
- shows a source discussion worth watching directly

But the page should still work without the media.

That is the important rule.

If the key idea only exists inside a video, the page is weaker for agents.

If the page explains the idea clearly in text and the video reinforces it, the page gets stronger.

## A practical standard to use

If you want a simple test, ask:

1. Can a human understand this page quickly?
2. Can an agent identify the main point in one pass?
3. Is there enough structure to reuse one section cleanly?
4. Is it obvious what the product does and how it works?
5. Are the docs or supporting pages easy to move through without guesswork?

If the answer is "not really" to several of those, the problem is not only SEO. The problem is that the site is hard to consume.

## The real lesson behind the YC line

"Make something agents want" is not really a slogan about agents.

It is a pressure test on clarity.

When agents become part of the discovery and implementation path, messy websites get exposed faster. Clean product surfaces get rewarded faster. Documentation stops being a side asset and becomes part of how demand is captured.

That is why this matters outside pure developer tools too.

If your site is supposed to educate, persuade, and get chosen in an AI-mediated environment, it has to be easy to read by both humans and machines.
